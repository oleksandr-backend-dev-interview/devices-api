# Design Decisions

## Identifier

* Devices use application-generated `UUID` primary keys.
* IDs are generated with `UUID.randomUUID()`, producing UUIDv4 values.
* This allows an identifier to be created without relying on a PostgreSQL sequence or identity column and supports independent ID generation by multiple application instances.
* UUIDv4 values have random insertion order. In a high-write production system, UUIDv7 could be evaluated because its time-ordered structure improves B-tree insertion locality and can reduce page splitting and index fragmentation.
* UUIDv7 was not introduced because Java 21 does not provide a standard UUIDv7 generator, and an additional dependency is not justified for this assignment.

## Device state

* Device state is persisted as `VARCHAR` using the enum constant names:

    * `AVAILABLE`
    * `IN_USE`
    * `INACTIVE`
* The database also contains a check constraint, so invalid states are rejected even when data is inserted outside the application.
* The check constraint is verified by a PostgreSQL integration test.
* The public API uses lowercase kebab-case values:

    * `available`
    * `in-use`
    * `inactive`
* Request-body deserialization and query-parameter conversion map these wire values to the corresponding Java enum constants.

## Creation time

* `creation_time` uses PostgreSQL `TIMESTAMPTZ`.
* The application represents creation time as `Instant`, and values are treated as absolute points in time.
* Creation time is generated by the server using an injected `Clock`.
* It cannot be supplied or modified through replacement or patch requests.
* The JPA column is marked `updatable = false`.

## Optimistic locking

* The `version` column is managed through JPA `@Version`.
* It detects conflicting concurrent persistence updates and prevents one transaction from silently overwriting another.
* Optimistic-lock failures are returned as HTTP `409 Conflict`.
* HTTP-level stale-client protection with ETags and `If-Match` is outside the scope of this assignment and would be a possible production improvement.

## Filtering and indexes

* Brand filtering is case-insensitive and uses exact matching.
* A functional PostgreSQL index on `LOWER(brand)` supports the corresponding case-insensitive equality query.
* State has a standard B-tree index because filtering by state is an explicit requirement.
* Since state has low cardinality, the usefulness of that index depends on real data distribution. In production, its use should be verified with representative data and `EXPLAIN ANALYZE`.
* Brand and state filters can be supplied independently or together.
* Explicit repository queries are used for the four possible filter combinations. A dynamic query framework such as JPA Specifications was not introduced because only two optional filters exist.

## Device creation

* A newly constructed device has state `AVAILABLE`.
* The create API accepts an optional state.
* When the state is omitted, the device remains `AVAILABLE`.
* When a state is supplied, it is applied explicitly after construction.

## Input normalization and validation

* Leading and trailing whitespace is removed from device names and brands.
* Whitespace-only and null values are rejected.
* Padding is considered accidental input rather than meaningful domain data.
* Bean validation rejects malformed create and replace requests before the controller invokes the service.
* The domain object independently protects its invariants, including nonblank values and restrictions applying to in-use devices.

## Replacement semantics

* `PUT` replaces all mutable fields: name, brand, and state.
* Identifier, creation time, and optimistic-lock version are not replaceable.
* A replace request must contain all mutable fields.
* When the current device is `IN_USE`, unchanged name and brand values are accepted, allowing a state-only replacement.
* A client cannot bypass the in-use restriction by changing the state and name or brand in the same request. Name and brand restrictions are evaluated while the device still has its original state.

## Partial-update semantics

* `PATCH` supports partial updates of name, brand, and state.
* Null-as-absent semantics are used: fields that are omitted or explicitly supplied as `null` are left unchanged.
* All current device fields are mandatory and cannot legitimately be cleared, so distinguishing omitted fields from explicit null values would not provide useful domain behavior.
* If nullable fields are introduced later, JSON Merge Patch or a tri-state representation such as `JsonNullable` should be considered.
* Null-skipping is handled in the service because field presence is an API transport concern.
* The entity retains explicit domain operations such as `rename`, `changeBrand`, and `changeState`.

## Domain operation restrictions

* Name and brand cannot be changed while the device’s current state is `IN_USE`.
* An `IN_USE` device cannot be deleted.
* State changes remain permitted for in-use devices so that they can be released or deactivated.
* Prohibited domain operations return HTTP `409 Conflict`.

## Error responses

* Errors use Spring `ProblemDetail` and the `application/problem+json` media type.
* Resource not found: `404 Not Found`.
* Invalid input, malformed JSON, invalid UUID, or unsupported state: `400 Bad Request`.
* Prohibited domain operation or optimistic-lock conflict: `409 Conflict`.
* Successful deletion: `204 No Content`.

## Architecture

* The application uses a straightforward layered structure with controller, service, domain, and repository responsibilities.
* A full hexagonal architecture was not introduced because the application has one HTTP interface, one persistence technology, and a small domain.
* Service interfaces with a single `Impl` implementation were avoided because they would add indirection without introducing a genuine architectural boundary.
* Spring Data repository interfaces remain appropriate because Spring supplies their implementations.

## Database migrations

* Flyway owns database schema creation and evolution.
* Hibernate is configured with `ddl-auto: validate`, so it verifies entity-to-schema compatibility without modifying the schema.
* The same migration is exercised locally, through Docker Compose, and in PostgreSQL Testcontainers tests.

## Testing strategy

* Domain unit tests verify device invariants independently of Spring.
* Service unit tests verify orchestration and repository interaction with Mockito.
* `@DataJpaTest` integration tests verify JPA mappings, Flyway schema compatibility, constraints, queries, and optimistic locking against real PostgreSQL.
* MVC slice tests verify validation, JSON mapping, query-parameter conversion, HTTP status codes, and `ProblemDetail` responses with a mocked service.
* A full end-to-end test starts the application on a random port and exercises the main device lifecycle through real HTTP and a PostgreSQL Testcontainer.
